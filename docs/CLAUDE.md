## ドキュメント選択ガイド

「どのフォルダを触るか」の判断基準です。

| ユーザーの依頼 | 触るべきフォルダ |
|---|---|
| 機能要件・プロダクト目標が変わった | `01-requirements` |
| システム構成・データモデル・API 設計が変わった | `02-basic-design` |
| コーディング規約・命名規則・フォルダ構成が変わった | `03-detailed-design` |
| 新しい開発作業を始める | `04-implementation/[frontend|backend]/` に新規フォルダを作成 |
| 将来やりたいことのメモ | `04-implementation/backlog.md` |
| Docker / PostgreSQL などのコマンド手順 | `04-implementation/runbooks/` |

## 永続的ドキュメント（01〜03）vs 作業単位ドキュメント（04）

| | 永続的ドキュメント | 作業単位ドキュメント |
|---|---|---|
| **場所** | `01-requirements/` 〜 `03-detailed-design/` | `04-implementation/[frontend\|backend]/[YYYYMMDD]-[title]/` |
| **内容** | 「このアプリは何か」「どう作るか」の定義 | 「今回の作業で何をするか」の記録 |
| **更新タイミング** | 設計・方針が根本的に変わるときのみ | 開発作業のたびに新規フォルダを作成 |
| **変更後の扱い** | 最新状態を維持 | 完了後も削除せず履歴として保持 |

## 作業単位ドキュメント

### 概要

特定の開発作業における「**今回何をするか**」を定義する一時的なドキュメント。
フロントエンド・バックエンドに応じて `frontend/` または `backend/` 配下に配置します。
作業完了後は参照用として保持されますが、新しい作業では新しいディレクトリを作成します。

### ディレクトリ命名

titleは作業単位の概要を簡潔にまとめたものにすること。

```
docs/04-implementation/[frontend|backend]/[YYYYMMDD]-[kebab-case-title]/
```

例：
- `docs/04-implementation/frontend/20260322-initial-implementation/`
- `docs/04-implementation/backend/20260328-add-backend/`

### 作成するファイルと順序

以下の順番で作成し、**1 ファイルごとにユーザーの確認・承認を得てから次に進むこと**。

1. `requirements.md` — 今回の作業の要求内容
    - 変更・追加する機能の説明
    - ユーザーストーリー
    - 受け入れ条件
    - 制約事項

2. `design.md`       — 実装アプローチと変更内容
    - 実装アプローチ
    - 変更するコンポーネント
    - データ構造の変更
    - 影響範囲の分析

1. `tasklist.md`     — 具体的な実装タスク
    - 具体的な実装タスク
    - タスクの進捗状況
    - 完了条件
2. `decision.md`     — （オプション）重要な決定事項の記録

### 禁止事項

- **過去の作業単位フォルダを修正しない。** 完了した作業の `requirements.md` / `design.md` / `tasklist.md` は意思決定の履歴記録であり、変更不可。
- **実装作業のたびに永続的ドキュメントを更新しない。** 01〜03 は設計変更時のみ更新する。
- **承認なしに複数ファイルを連続作成しない。** 必ず 1 ファイルごとに確認を取る。
