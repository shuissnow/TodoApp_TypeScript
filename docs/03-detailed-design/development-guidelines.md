## 1. Git規約

### ブランチ戦略

```
main                    # 本番環境（直接コミット禁止）
└── feature/[機能名]    # 機能開発ブランチ
└── fix/[修正内容]      # バグ修正ブランチ
└── docs/[ドキュメント] # ドキュメント更新ブランチ
```

### ブランチ命名規則

| 種別 | 形式 | 例 |
|------|------|----|
| 機能追加 | `feature/[機能名]` | `feature/add-filter` |
| バグ修正 | `fix/[修正内容]` | `fix/empty-input-validation` |
| ドキュメント | `docs/[内容]` | `docs/update-readme` |

### コミットメッセージ規約

[Conventional Commits](https://www.conventionalcommits.org/) に準拠する。

```
<type>: <概要>

[本文（任意）]
```

| type | 用途 |
|------|------|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | フォーマット変更（動作に影響しない） |
| `refactor` | リファクタリング |
| `test` | テストの追加・修正 |
| `chore` | ビルド・ツール設定の変更 |

**例：**
```
feat: タスクのフィルタリング機能を追加

すべて・未完了・完了済みの3種類のフィルターを実装
フィルター状態はURLクエリパラメータには保存しない
```

### プルリクエスト

- `main` へのマージは必ずPRを経由する
- セルフレビュー後にマージしてよい（v1.0は1人開発のため）
- マージ方法: **Squash and merge**

---

## 2. セキュリティ規約

- シークレット情報（APIキー・パスワード等）はソースコードに直接書かない。環境変数または設定ファイルを使用する
- `.env` ファイルは `.gitignore` に追加し、リポジトリにコミットしない
- ユーザー入力は必ずバリデーションを行う
