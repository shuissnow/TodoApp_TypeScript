# ドキュメント管理

このフォルダは、TodoApp の開発に関わるすべてのドキュメントを管理する場所です。

ドキュメントは **SDLC（ソフトウェア開発ライフサイクル）のフェーズ順** に番号付きフォルダで整理されています。番号を見ればどのフェーズのドキュメントか一目でわかり、プロジェクト全体の流れを把握しやすくなっています。

## 各フォルダの説明

### 01-requirements — 要件定義

「このプロダクトは何のために存在するのか」を定義するドキュメントです。機能の追加・変更を検討するとき、まずここに立ち返って判断の軸を確認してください。

| ファイル | 内容 |
|---|---|
| `product-requirements.md` | プロダクトビジョン、ターゲットユーザー、機能要件、受け入れ条件 |

---

### 02-basic-design — 基本設計

システム全体の構造と技術的な方針を定義するドキュメントです。「どういう仕組みで作るか」の大枠を示します。実装方針に迷ったときはここを参照してください。

| ファイル | 内容 |
|---|---|
| `functional-design.md` | 機能設計書（データモデル、画面遷移、API設計など） |
| `architecture.md` | 技術仕様書（技術スタック、インフラ構成、パフォーマンス要件） |

---

### 03-detailed-design — 詳細設計

実装に直結する詳細な仕様を定義するドキュメントです。コードを書く前に必ず確認し、命名規則やフォルダ構成のルールを守って開発を進めてください。

| ファイル | 内容 |
|---|---|
| `repository-structure.md` | リポジトリ全体のフォルダ・ファイル構成 |
| `repository-structure-frontend.md` | フロントエンドの構造・命名規則・環境変数 |
| `repository-structure-backend.md` | バックエンドの構造・命名規則・環境変数 |
| `development-guidelines.md` | コーディング規約、Git規約、レビュー基準 |
| `glossary.md` | プロジェクト内で使う用語の定義（ユビキタス言語） |

---

### 04-implementation — 実装

実際の開発作業に関わるドキュメントをまとめたフォルダです。作業単位のドキュメント（要件・設計・タスクリスト）は `frontend/` または `backend/` に、日付と作業名でフォルダを作って管理します。

**作業単位フォルダの構成例：**

```
04-implementation/
├── frontend/
│   └── 20260322-initial-implementation/
│       ├── requirements.md   # この作業で何をするか
│       ├── design.md         # どう実装するか
│       └── tasklist.md       # 具体的なタスクと進捗
└── backend/
    └── 20260328-add-backend/
        ├── requirements.md
        ├── design.md
        └── tasklist.md
```

`runbooks/` には Docker や PostgreSQL などのコマンド手順書を配置しています。環境構築や操作手順で迷ったときに参照してください。

`backlog.md` は将来的に取り組みたいことのメモです。優先度が決まった時点でタスク化します。

## 新しいメンバーへ

プロジェクトを理解するには、以下の順番でドキュメントを読むことをお勧めします。

1. `01-requirements/product-requirements.md` — まずプロダクトの目的を把握する
2. `02-basic-design/architecture.md` — 技術スタックと全体構成を把握する
3. `02-basic-design/functional-design.md` — 機能の設計を理解する
4. `03-detailed-design/development-guidelines.md` — 開発ルールを確認する
5. `03-detailed-design/glossary.md` — 用語を確認する

---

## ドキュメントの更新ルール

### 永続的ドキュメント（01〜03）
プロダクトの基本設計を記述するドキュメントです。頻繁に変更されるものではありません。**機能の追加・設計の変更があった場合のみ**、該当するドキュメントを更新してください。

### 作業単位ドキュメント（04）
特定の開発作業ごとに作成するドキュメントです。作業が始まるたびに新しいフォルダを作成し、作業完了後もそのまま履歴として残します。過去の作業の意思決定を振り返るための記録になります。

### テストドキュメント（05〜07）
テスト方針や設計が固まった時点で作成・更新します。
