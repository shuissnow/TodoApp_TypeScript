## 設定ファイル・Docker の変更ルール

### 操作禁止

**以下のフォルダは絶対に編集・削除しないこと。ビルドツールが自動生成・管理するフォルダである。**

- `dist/` — `npm run build` によって生成されるビルド成果物
- `node_modules/` — `npm install` によってインストールされるパッケージ群

### 承認ルール

**以下のファイルはすべて変更前に承認を得ること。**

- `Dockerfile`
- `package.json`
- `vite.config.ts`
- `tsconfig.json`
- `eslint.config.ts`
- `nginx.conf`
- `index.html`

---

### ファイルごとの注意点

#### `Dockerfile`

- マルチステージビルド構成（`builder`ステージ: 開発用 Vite, 最終ステージ: 本番用 nginx）を維持すること
- ベースイメージのバージョンを変更する場合は、既存の依存関係との互換性を確認すること
- `builder` ステージと最終ステージの不整合（コピーパスのズレなど）に注意すること

#### `package.json`

- パッケージを追加する前に、同等の機能を持つパッケージが既にインストールされていないか確認すること
- アプリ実行に必要なものは `dependencies`、開発・ビルドのみに必要なものは `devDependencies` に分類すること
- バージョン指定は既存の記法（`^` など）に合わせること

#### `vite.config.ts`

- プロキシ設定を変更すると、フロントエンドからバックエンド API への接続に影響する
- ビルド出力先（`build.outDir`）を変更すると、`Dockerfile` の `COPY` パスとズレる可能性がある
- プラグインの追加・削除は `package.json` への変更も伴う

#### `tsconfig.json`

- `strict` 関連オプションを変更すると、既存コード全体に型エラーが波及する可能性がある
- パスエイリアス（`paths`）を追加・変更する場合は、`vite.config.ts` の `resolve.alias` とも整合させること

#### `eslint.config.ts`

- ルールを無効化・緩和する変更は原則として行わないこと
- ルールを追加・強化する場合は、既存コード全体への影響を確認すること

#### `nginx.conf`

- SPA のルーティングに必要な `try_files` の設定を削除しないこと（削除するとリロード時に 404 になる）
- 静的アセットのキャッシュ設定を変更する場合は、本番環境での影響を考慮すること

#### `index.html`

- Vite のエントリーポイントとなる `<script type="module">` タグを削除しないこと
- アプリのマウント先となる `<div>` 要素を削除しないこと

#### `public/`

- Vite によるバンドル処理を受けず、そのまま配信される（ファビコン・SVG アイコンなど）
- 大きなファイルを追加するとロード時間に影響するため注意すること
