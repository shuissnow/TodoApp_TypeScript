## 1. Web APIとは何か

**API（Application Programming Interface）** = 「こういうリクエストを送れば、こういうデータを返す」という約束事。

```
フロントエンド                  バックエンド(Web API)
(ブラウザ)                      (C# ASP.NET Core)
    │                               │
    │── GET /api/todos ──────────→  │ タスク一覧を取れ
    │                               │ DBを調べる...
    │← [{"id":"...", ...}] ─────── │ JSON形式で返す
```

普通のC#プログラムと違い、**HTTPという通信プロトコル**でやり取りします。

## 2. HTTPの基礎

### HTTPメソッド（何をしたいか）

| メソッド | 意味 | このプロジェクトでの使用例 |
|---------|------|--------------------------|
| `GET` | データを取得 | タスク一覧を取得 |
| `POST` | 新規作成 | タスクを新規作成 |
| `PUT` | 更新 | タスクを編集・完了切替 |
| `DELETE` | 削除 | タスクを削除 |

### HTTPステータスコード（結果を数字で表す）

| コード | 意味 | いつ返すか |
|--------|------|-----------|
| `200 OK` | 成功 | GETで一覧取得成功 |
| `201 Created` | 作成成功 | POSTでタスク作成成功 |
| `204 No Content` | 成功（返すデータなし） | DELETE成功 |
| `400 Bad Request` | リクエストが不正 | textが空でPOSTされた |
| `404 Not Found` | 見つからない | 存在しないIDのタスクをPUT/DELETE |
| `500 Internal Server Error` | サーバー側エラー | DBが落ちている等 |

## 3. ASP.NET Core Web APIの構造

### レイヤー構成

```
HTTPリクエスト
    ↓
[Controller]        ← URLを受け取り、レスポンスを返す（交通整理係）
    ↓ DTOs           ← リクエスト/レスポンスのデータ形式
[Service]           ← ビジネスロジック（「textが空ならエラー」など）
    ↓ Entities       ← DBのテーブルに対応するクラス
[Repository]        ← DBへの読み書き
    ↓
[AppDbContext]      ← EF Core（DBとのブリッジ）
    ↓
PostgreSQL
```

### Controller — HTTPの入口

URLとHTTPメソッドに対応したメソッドを定義します。ロジックは持たず、受け取ったデータをServiceに渡してレスポンスを返すだけです。

```csharp
// Controllers/TodosController.cs
[ApiController]           // 「これはWeb APIのコントローラーです」という宣言
[Route("api/todos")]      // URLの /api/todos をここで処理する
public class TodosController : ControllerBase
{
    private readonly ITodoService _service;

    public TodosController(ITodoService service)
    {
        _service = service;
    }

    [HttpGet]             // GET /api/todos
    public async Task<IActionResult> GetAll()
    {
        var todos = await _service.GetAllAsync();
        return Ok(todos); // 200 OK + JSONを返す
    }

    [HttpPost]            // POST /api/todos
    public async Task<IActionResult> Create(CreateTodoRequest request)
    {
        var todo = await _service.CreateAsync(request);
        return CreatedAtAction(nameof(GetAll), todo); // 201 Created
    }

    [HttpDelete("{id}")]  // DELETE /api/todos/{id}
    public async Task<IActionResult> Delete(Guid id)
    {
        var success = await _service.DeleteAsync(id);
        if (!success) return NotFound(); // 404
        return NoContent();              // 204
    }
}
```

### DTO — リクエスト/レスポンスのデータ形式

フロントエンドから受け取るJSON、またはフロントエンドへ返すJSONの形を定義します。

```csharp
// DTOs/CreateTodoRequest.cs
// POST /api/todos のボディ: { "text": "買い物" }
public class CreateTodoRequest
{
    public string Text { get; set; } = string.Empty;
}

// DTOs/UpdateTodoRequest.cs
// PUT /api/todos/{id} のボディ
public class UpdateTodoRequest
{
    public string? Text { get; set; }
    public bool? Completed { get; set; }
}
```

**DTOとEntityを分ける理由：**

- **Entity**（`Todo.cs`）= DBのテーブルそのもの。`CreatedAt` などの内部フィールドも持つ
- **DTO** = 外部に公開する形。「textだけ受け取ればいい」という場合はDTOを使う
- Entityをそのまま外部に公開するとDBの詳細が漏れる

### Service — ビジネスロジック

バリデーションや業務ルールを担当します。

```csharp
// Services/TodoService.cs
public class TodoService : ITodoService
{
    private readonly ITodoRepository _repository;

    public async Task<Todo> CreateAsync(CreateTodoRequest request)
    {
        // バリデーション（ビジネスロジック）
        if (string.IsNullOrWhiteSpace(request.Text))
            throw new ArgumentException("textは必須です");

        var todo = new Todo
        {
            Id = Guid.NewGuid(),
            Text = request.Text,
            Completed = false,
            CreatedAt = DateTime.UtcNow
        };

        return await _repository.AddAsync(todo);
    }
}
```

### Repository — DBアクセス

EF Coreを直接触るのはここだけ。DBのCRUD操作をカプセル化します。

```csharp
// Repositories/TodoRepository.cs
public class TodoRepository : ITodoRepository
{
    private readonly AppDbContext _context;

    public async Task<List<Todo>> GetAllAsync()
    {
        // EF CoreでSQLを書かずにDBから取得
        return await _context.Todos
            .OrderBy(t => t.CreatedAt)
            .ToListAsync();
    }

    public async Task<Todo> AddAsync(Todo todo)
    {
        _context.Todos.Add(todo);
        await _context.SaveChangesAsync(); // ここでINSERT文が実行される
        return todo;
    }
}
```

### AppDbContext — EF CoreのDBブリッジ

```csharp
// Data/AppDbContext.cs
public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    // このプロパティがDBの "todos" テーブルに対応する
    public DbSet<Todo> Todos { get; set; }
}
```

## 4. 依存性注入（DI）

ASP.NET Coreで最も重要な概念の一つ。`new` でオブジェクトを作らず、フレームワークが必要なものを自動で組み立てて渡してくれる仕組みです。

### 登録（Program.cs）

```csharp
builder.Services.AddScoped<ITodoService, TodoService>();
builder.Services.AddScoped<ITodoRepository, TodoRepository>();
// 「ITodoServiceが必要なときはTodoServiceを作れ」と教えている
```

### 動作イメージ

```
リクエストが来る
    ↓
ASP.NET が TodosController を作ろうとする
    ↓
「コンストラクタに ITodoService が必要だな」
    ↓
登録を見て「TodoService を作れ」
    ↓
「TodoService には ITodoRepository が必要だな」
    ↓
「TodoRepository を作れ」
    ↓
全部組み立てて TodosController に渡す
```

**インターフェース（`ITodoService`）を使う理由：**
- テスト時に「本物のDB」ではなく「偽物のDB」を差し込める
- 実装を後で変えても、使う側のコードを変えなくていい

## 5. Program.cs — アプリの起動設定

```csharp
var builder = WebApplication.CreateBuilder(args);

// サービスの登録（DI設定）
builder.Services.AddControllers();
builder.Services.AddDbContext<AppDbContext>(
    options => options.UseNpgsql(
        builder.Configuration.GetConnectionString("DefaultConnection")
    )
);
builder.Services.AddScoped<ITodoService, TodoService>();
builder.Services.AddScoped<ITodoRepository, TodoRepository>();
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowFrontend", policy =>
    {
        policy.WithOrigins("http://localhost:5173")
              .AllowAnyHeader()
              .AllowAnyMethod();
    });
});

var app = builder.Build();

// ミドルウェアの設定（リクエストが通る順番）
app.UseCors("AllowFrontend");
app.UseRouting();
app.MapControllers();

app.Run();
```

## 6. リクエストが届いてから返るまでの全体の流れ

例：フロントエンドが `POST /api/todos` を送った場合

```
1. フロントエンド
   fetch("http://localhost:8080/api/todos", {
     method: "POST",
     body: JSON.stringify({ text: "買い物" })
   })

2. ASP.NET Core がリクエストを受け取る
   → URLとメソッドを見て TodosController.Create() を呼ぶ

3. [HttpPost] が付いた Create メソッドが実行
   → JSONボディを自動で CreateTodoRequest に変換してくれる

4. _service.CreateAsync(request) を呼ぶ

5. TodoService.CreateAsync()
   → バリデーション
   → Todo エンティティを作成
   → _repository.AddAsync(todo) を呼ぶ

6. TodoRepository.AddAsync()
   → EF Core が SQL の INSERT を自動生成
   → PostgreSQL に保存

7. 保存されたTodoがControllerに戻る

8. return CreatedAtAction(...) で 201 + JSONをフロントに返す
```

## 7. EF Core マイグレーション

C#のクラス定義からDBのテーブルを自動生成する仕組みです。

```bash
# マイグレーションファイルを生成
dotnet ef migrations add InitialCreate

# 生成されたSQLをDBに適用（テーブル作成）
dotnet ef database update
```

このプロジェクトでは起動時に自動実行します：

```csharp
// Program.cs
using var scope = app.Services.CreateScope();
var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
db.Database.Migrate(); // 起動時にマイグレーションを自動適用
```

## 8. まとめ

| 概念 | 役割 | 例 |
|------|------|-----|
| Controller | HTTPの入口・出口 | URLとメソッドを見て処理を振り分ける |
| DTO | データの受け渡し形式 | `{ "text": "買い物" }` |
| Service | ビジネスロジック | textが空ならエラー |
| Repository | DBアクセス | EF CoreでSELECT/INSERT/DELETE |
| AppDbContext | EF CoreのDB接続 | C#クラス ↔ DBテーブルの対応 |
| DI | オブジェクトの自動生成 | `new` せずに必要なものが渡ってくる |
| Program.cs | 起動設定 | 全サービスの登録・ミドルウェア設定 |
